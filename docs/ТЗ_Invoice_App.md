**Версия:** 1.2

**Источник данных:** [Google Sheet «Копия Invoice»](https://docs.google.com/spreadsheets/d/1fI2ogMNkTRs9gOmxDl7TPkXyhTLtjn9zhJW9VBYy1Q4/)

## 1. Цель проекта

Заменить существующий процесс ведения инвойсов в Google-таблице на полноценное веб-приложение. Приложение должно позволять:
- вести справочники своих компаний и контрагентов в БД;
- формировать инвойсы/receipts по нескольким шаблонам (THB, USD, Crypto, Cash);
- предоставлять предпросмотр документа в том виде, в каком он будет напечатан;
- генерировать и скачивать PDF;
- разграничивать доступ между супер-админом и обычным пользователем.

## 3. Структура документа «Инвойс»

Единая модель инвойса, собранная из всех шаблонов:

### 3.1. Шапка
- Логотип нашей компании (один на компанию — см. §5.1)
- Заголовок: **INVOICE** / **RECEIPT**
- `Invoice №` — уникальный номер (формат по ТЗ: `ДД/ММ/ГГГГ-NNN` — как в таблице, например `27/11/2025-0007`)
- `Date` — дата выписки (по умолчанию = сегодня)
- `Date (other)` / `Due Date` — альтернативная дата/срок оплаты
- `Payment Terms` — способ оплаты (Bank Transfer / SWIFT Transfer / Crypto Transfer)
- `BALANCE DUE` / `BALANCE PAID` — итоговая сумма (для Receipt — другая надпись)

### 3.2. Блок «От кого» (Our company)
Автоподстановка из справочника компаний при выборе:
- Название компании
- Адрес
- Tax ID
- Телефон
- E-mail
- **Банковские реквизиты:** Bank Name, Account Name, Account Number, Swift code, Bank Branch, Bank Address (для офшоров)

### 3.3. Блок «Кому» (Bill To / Counterparty)
Автоподстановка из справочника контрагентов:
- Название
- Адрес
- Tax ID
- Телефон
- E-mail

### 3.4. Таблица позиций
Колонки во вводе:
| № | Item (Commission / Bonus) | Project | Unit № | Selling price | % commission | Bonus | Selling price correction | Commission correction | Note |
|---|---|---|---|---|---|---|---|---|---|

Колонки в выводе (печатная форма):
| Item | Quantity | Rate | Amount |

Логика расчёта:
- Если выбран **Commission**: `Amount = (Selling price + Selling price correction) × % commission + Commission correction`
- Если выбран **Bonus**: `Amount = Bonus`
- Строки с полем `Note` добавляют пояснение о корректировке расчёта.
- Количество позиций — произвольное; пустые строки скрываются.

### 3.5. Итоги
- `Subtotal` = сумма всех `Amount`
- `Withholding Tax (3%)` = применяется, если чекбокс включён — `Subtotal × 0.03`
- `VAT (7%)` = применяется, если чекбокс включён — `Subtotal × 0.07`
- **Total / Balance Due** = `Subtotal + VAT − Withholding Tax`

### 3.5.1. Работа с валютами (THB / USD)
Основная валюта ввода — **THB**. Для добавления пересчёта в USD:
- в форме создания инвойса есть чекбокс **«Показать сумму в USD»**;
- если чекбокс **снят** — инвойс только в THB, USD-блок не выводится;
- если чекбокс **установлен**:
  - система автоматически запрашивает актуальный курс THB→USD у Wise API в момент создания/пересчёта;
  - в инвойсе дополнительно печатается строка с USD-суммой и зафиксированным курсом (`Rate: 1 USD = X.XX THB, Wise, <дата>`);
  - курс сохраняется в карточке инвойса, чтобы при повторной генерации PDF сумма не менялась.
- Кнопка «Обновить курс» позволяет пересчитать USD в ручном режиме до выпуска.

Для шаблона **Invoice USD** (офшор) — раскладка `Subtotal / Fee / VAT / Net Amount` без тайских налогов.

### 3.6. Notes / Payment instruction
- Блок «The payment by bank transfer to:» с реквизитами нашей компании (дублирует §3.2);
- Текстовое описание: например `1% Commission for sold out Punyisa Plot 40`.

## 4. Функциональные требования

### 4.1. Управление справочниками
- **Компании (наши)** — CRUD:
  - все поля из §3.2 + логотип + юридическая/налоговая принадлежность (резидент / офшор, влияет на шаблон);
  - несколько наборов банковских реквизитов на одну компанию (если нужно платить в разных валютах);
  - проставляется признак «активна» (soft-delete).
- **Контрагенты** — CRUD:
  - все поля из §3.3;
  - импорт полного списка из Google-таблицы при старте;
  - **во всех выпадающих списках контрагентов — поиск по подстроке** (type-ahead, совпадение по 2+ символам, нечувствительный к регистру, ищет и по латинице, и по тайскому).
- **Проекты и юниты (Unit №)** — в MVP: свободный ввод текстом (как сейчас). На следующем этапе — отдельный справочник проектов со списком юнитов и подстановкой.
- **Типы позиций** — Commission, Bonus (возможность добавления новых).
- **Способы оплаты** — Bank Transfer, SWIFT Transfer, Crypto Transfer (возможность добавления новых).
- **Валюты** — минимум THB, USD; архитектурно закладываем расширяемость (EUR, RUB, USDT и т.д.).

### 4.2. Создание инвойса («мастер» в 4 шага)
1. **Параметры:** дата, способ оплаты, валюта → определяет шаблон (IBG / IB Group THB / USD / Crypto / Receipt).
2. **Стороны:** выбор «Our company» из списка (с автозаполнением), выбор «Counterparty» (с автозаполнением или быстрое создание нового).
3. **Позиции:** добавление 1+ строк (Commission / Bonus) с расчётом сумм «на лету»; корректировки Selling price и Commission; чекбоксы VAT / Withholding Tax.
4. **Preview + генерация:** просмотр готового PDF 1:1 с текущей печатной формой Google Sheets → кнопки «Скачать PDF», «Сохранить черновик», «Выпустить инвойс».

### 4.3. Нумерация инвойсов
- Формат: `ДД/ММ/ГГГГ-NNNN` (как сейчас в таблице, напр. `27/11/2025-0007`);
- **NNNN — сквозной счётчик по всей системе** (общий на все наши компании);
- Номер генерируется автоматически при создании черновика;
- Супер-админ может переопределить номер вручную;
- Дубли номеров — запрещены на уровне БД (`UNIQUE`).

### 4.4. Жизненный цикл инвойса и автогенерация Receipt
Состояния: `Draft` → `Issued` → `Paid` / `Cancelled`.
- `Issued` блокирует изменение большинства полей; правки — только через «воид + новый».
- **Отметка «Paid» бухгалтером** (кнопка «Отметить оплаченным» в карточке):
  1. фиксируется дата фактической оплаты и кто отметил (audit log);
  2. **система автоматически генерирует Receipt** на основе данных инвойса (шапка `RECEIPT`, подпись `BALANCE PAID` вместо `BALANCE DUE`);
  3. сгенерированный Receipt сохраняется в той же таблице, что и Invoice, но с `type = receipt` и ссылкой на родительский инвойс (`parent_invoice_id`);
  4. у Receipt — собственный публичный URL на PDF (см. §4.5);
  5. нумерация Receipt = номер инвойса с суффиксом `-R` (напр. `27/11/2025-0007-R`), либо отдельная сквозная — уточним на этапе 0.
- Отмена (`Cancelled`) возможна только для `Draft`/`Issued` и только супер-админом, с обязательной причиной.

### 4.5. Предпросмотр, PDF и хранение
- **Preview на экране** — HTML-рендер, точно повторяющий текущий шаблон Google Sheets (A4, portrait).
- **PDF:**
  - формат A4, ориентация portrait;
  - имя файла: `Invoice_<company>_<№>.pdf` (или `Receipt_...` для receipt);
  - шрифты с поддержкой кириллицы и тайского (Noto Sans / Sarabun) — в таблице встречается тайский текст (напр. `ซอยริมคลอง...`).
- **Хранение PDF (обязательно):**
  - PDF сохраняется в объектном хранилище S3 (MinIO self-hosted или AWS S3 / Backblaze B2);
  - в БД, в таблице инвойсов, хранится поле `pdf_url` — постоянная публичная ссылка на файл;
  - у каждого инвойса и у каждого Receipt — **своя** ссылка на PDF;
  - ссылка доступна из карточки в UI (кнопка «Открыть PDF») и может быть скопирована;
  - при перевыпуске документа (правка супер-админом) сохраняются **все версии** PDF (`pdf_url` — актуальная, плюс история в `pdf_versions[]`);
  - бэкап PDF-хранилища — ежедневно, вместе с БД.
- **Ссылки:**
  - публичная ссылка с токеном (долгая, не угадывается) — без авторизации, удобно переслать контрагенту;
  - внутренняя ссылка приложения (`/invoices/<id>`) — требует авторизации.
- **Массовый экспорт** (zip с пачкой PDF) — для супер-админа, по фильтру (период, компания, статус).

### 4.7. Аудит
Журнал событий: кто создал/изменил/выпустил/отменил инвойс, когда, что именно изменилось (diff).

### 4.8. Интеграция с Wise (курс валют)
- Источник курса: **Wise API** (https://wise.com/developers — endpoint `v1/rates`);
- авторизация: API key хранится в env-переменных сервера;
- кеш курса в Redis с TTL 10 минут, чтобы не дёргать Wise на каждое нажатие;
- если Wise недоступен — показываем последний известный курс + предупреждение, и даём возможность ввести курс вручную (только супер-админу);
- при выпуске инвойса (`Issued`) курс фиксируется навсегда в карточке (`exchange_rate`, `exchange_rate_source`, `exchange_rate_at`).

> 📌 На будущее: интеграция с Telegram-ботом для выгрузки PDF контрагентам (см. §9, вопрос 4) — проектируется отдельно.

### 4.9. Миграция данных из Google Sheets
Однократный импорт:
- всех компаний из выпадающего списка → справочник «Наши компании»;
- всех контрагентов из полей Counterparty листов Info → справочник «Контрагенты»;
- всех существующих инвойсов (листы Invoice*) → таблица инвойсов со статусом `Issued`.

Выполняется скриптом, запускается через админ-панель с предпросмотром и возможностью откатить.

### 5.2. Индексы/ограничения
- `UNIQUE(invoice.number)` — защита от дублей.
- `invoice.status` — только движение «вперёд» (Draft → Issued → Paid/Cancelled), без отката.
- `invoice.parent_invoice_id` — индекс для быстрого поиска receipt по инвойсу.
- Полнотекстовый индекс на `counterparty.name` (включая тайский) для type-ahead поиска.

### 5.3. Хранение файлов
- S3-bucket `invoices/` — структура `year=YYYY/month=MM/<invoice_id>/v<N>.pdf`;
- публичная ссылка: `https://cdn.<domain>/i/<invoice_id>?t=<signed_token>` (токен живёт 10 лет, непредсказуемый);
- backup: ежедневный snapshot S3 + БД.

## 6. Нефункциональные требования

- **Производительность:** создание PDF ≤ 2 сек на документ; страница со списком инвойсов ≤ 1 сек при 10k записей.
- **Надёжность:** ежедневный бэкап БД; мягкое удаление (soft-delete) для всех справочников.
- **Безопасность:**
  - HTTPS обязательно;
  - пароли — bcrypt / argon2;
  - RBAC на уровне API;
  - rate-limit на авторизацию;
  - CSRF-защита.
- **Локализация UI:** русский (основной), английский (как минимум для печатных форм — они всегда на английском).
- **Поддержка шрифтов:** тайский, кириллица, латиница.

### 9.1. Второй раунд уточнений — согласовано

| # | Вопрос | Решение |
|---|---|---|
| 1 | Реестр «наших компаний» | **9 компаний** собраны в отдельном файле [`Наши_компании.md`](./Наши_компании.md). Часть полей (Tax ID, адреса, Branch) уточняется при деплое. Крипто-счёт IBG Property и Miss Jana Astakhova в реестр **не входят** |

| 5 | Выдача PDF контрагенту в MVP | **Вариант C:** в карточке и инвойса, и receipt — **две кнопки**: «Скачать PDF» и «Скопировать публичную ссылку» |

### 9.2. Следствия для реализации

**По номеру Receipt (вопрос 2).**
- Валидация уникальности: `UNIQUE(invoice.number)` продолжает работать — `27/11/2025-0007` и `27/11/2025-0007-R` это разные строки.
- При генерации Receipt система подставляет номер автоматически; супер-админ может его переопределить.

**По USD-комиссии (вопрос 3).**
- Поля ввода в шаблоне «Invoice USD»:
  - `Selling price (THB)` — обязательно;
  - `% commission` — обязательно;
  - `Rate (Wise)` — подтягивается автоматически, но **блокируется** после выпуска (`Issued`);
  - `Commission (THB)` — расчёт: `Selling price × %`;
  - `Commission (USD)` — расчёт: `Commission (THB) / Rate`;
  - В печатной форме показывается и THB, и USD, с фиксацией курса в Notes (`Rate: 1 USD = X.XX THB, Wise, <дата>`).
- Это соответствует текущей практике в Google-таблице (лист `Invoice USD`: `฿886,296.40 × 7% = $62,040.75`).

**По выдаче PDF (вопрос 5).**
- В карточке документа (и Invoice, и Receipt) — две кнопки:
  - **«Скачать PDF»** → отдаёт файл пользователю напрямую;
  - **«Скопировать ссылку»** → копирует в буфер публичный URL (`https://cdn.<domain>/i/<id>?t=<token>`), показывает тост-уведомление.
- Дополнительно — кнопка «Открыть в новой вкладке» для предпросмотра без скачивания.
