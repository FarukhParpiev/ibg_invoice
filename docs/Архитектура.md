**Связанные документы:** [ТЗ_Invoice_App.md](./ТЗ_Invoice_App.md), [Наши_компании.md](./Наши_компании.md)

## ✅ Зафиксированный стек (решение заказчика — упрощённый MVP)

**Стратегия:** всё внутри Vercel, одна точка входа — один аккаунт для биллинга, одна консоль управления. Отдельных регистраций на сторонних сервисах не нужно.

| Компонент | Сервис | Как подключается |
|---|---|---|
| Frontend + Backend API | **Vercel** (Next.js 14 + API Routes + Server Actions) | Уже есть аккаунт |
| База данных | **Neon Serverless Postgres** через Vercel Marketplace | Подключается из dashboard Vercel в один клик, биллинг — через Vercel |
| Кеш + очереди | **Upstash for Redis** через Vercel Marketplace | Аналогично, один клик |
| Объектное хранилище (PDF, логотипы) | **Vercel Blob** | Встроено в Vercel |
| Код | **GitHub** (нужна регистрация) | Интеграция Vercel ↔ GitHub — автодеплой |
| Мониторинг | отложено | Добавим в prod после MVP |
| Домен | временный `*.vercel.app` | Миграция на ваш — позже |

### Почему так
- Vercel Marketplace — это официальная интеграция Vercel с Neon, Upstash и другими. Ресурс создаётся прямо из dashboard Vercel, биллинг учитывается в вашем Vercel-аккаунте.
- Никаких отдельных паролей и ключей управлять не нужно.
- Все бесплатные лимиты покрывают MVP (до 100+ инвойсов в месяц).
- Переезд на отдельные аккаунты Neon/Upstash/Backblaze — это смена DSN в env-переменных, код не трогаем.

### Архитектурное упрощение
- **Единый Next.js-проект** (без отдельного NestJS): API Routes + Server Actions.
- **PDF-генерация:** `@sparticuz/chromium` + Puppeteer-core в serverless-функции Vercel.
- **Очереди:** QStash (от Upstash) через Vercel Marketplace для фоновых задач типа массового экспорта PDF.

## 2. Что где хранится

| Данные | Где хранятся | Почему именно там |
|---|---|---|
| Пользователи, компании, контрагенты, инвойсы, позиции, audit log | **PostgreSQL 16** | Транзакции, целостность, сложные запросы |
| Сессии авторизации, JWT-refresh токены | **PostgreSQL** + Redis (кеш) | Персистентные сессии, быстрое чтение |
| Кеш курсов Wise, rate-limit счётчики, очередь генерации PDF | **Redis 7** | В памяти, без IO-оверхеда |
| **Файлы PDF** (инвойсы, receipts, все версии) | **MinIO (S3-compatible)** | Объектное хранилище, публичные ссылки, дешёвый бэкап |
| Логотипы компаний | **MinIO** | То же, что и PDF |
| Логи приложения | **Loki** (или CloudWatch/папка с ротацией) | Поиск по логам, история 30 дней |
| Бэкапы | **Off-site S3** (Backblaze B2 / AWS S3 в другом регионе) | Защита от потери основного сервера |

## 3. Серверная инфраструктура

### 3.1. Рекомендованная конфигурация (production)

**Вариант A — self-hosted (рекомендуется, полный контроль):**

| Сервер | Роль | Размер | Провайдер |
|---|---|---|---|
| **App-1** | Next.js + NestJS + MinIO + Nginx + Redis | 4 vCPU / 8 GB RAM / 80 GB SSD | Hetzner CCX13 / DigitalOcean |
| **DB-1** | PostgreSQL primary | 2 vCPU / 4 GB RAM / 100 GB SSD | Hetzner CPX21 |
| **DB-2** (опционально) | PostgreSQL standby (репликация) | 2 vCPU / 4 GB RAM / 100 GB SSD | тот же провайдер, другая зона |

- Стоимость: ~€25–40/мес на MVP, можно вырастить до €100+/мес при росте.
- Если MVP — всё можно упаковать на **один** VPS (4 vCPU / 8 GB / 80 GB), разделить потом.

**Вариант B — managed (быстрее стартовать):**

| Сервис | Роль |
|---|---|
| **Vercel** | Next.js frontend (автодеплой, CDN) |
| **Railway / Fly.io** | NestJS backend |
| **Neon / Supabase Postgres** | Managed PostgreSQL с автобэкапом и репликацией |
| **Upstash Redis** | Managed Redis |
| **Backblaze B2 / AWS S3** | Объектное хранилище |
| **Cloudflare** | DNS, WAF, CDN |

- Плюсы: минимум работы с инфрой, SLA провайдера;
- Минусы: дороже на росте, меньше контроля, вендор-лок.

**Моя рекомендация:** стартовать на **Варианте B для MVP** (быстро и с SLA из коробки), потом при необходимости переехать на self-hosted. Модель данных и Docker-образы — идентичны, миграция — за 1–2 дня.

### 3.2. Как работает запрос (happy path)

1. Пользователь открывает `https://invoice.yourdomain.com/invoices/new`.
2. DNS Cloudflare → IP сервера.
3. Cloudflare проверяет DDoS, WAF, кеш (для статических файлов).
4. Nginx принимает запрос, проверяет SSL, роутит:
   - `/` → Next.js (SSR),
   - `/api/*` → NestJS.
5. Next.js запрашивает у NestJS данные (JSON API).
6. NestJS идёт в PostgreSQL через Prisma, получает данные.
7. Ответ возвращается пользователю.
8. Пользователь жмёт «Выпустить» → NestJS:
   - пишет в БД `status = 'issued'`,
   - кладёт задачу в Redis-очередь BullMQ (`generate-pdf`);
9. Worker подхватывает задачу:
   - рендерит HTML-шаблон в Puppeteer,
   - сохраняет PDF в MinIO,
   - обновляет `invoice.pdf_url` в БД;
10. Пользователь получает уведомление «PDF готов».

### 3.3. Как происходит деплой

1. Разработчик пушит в `main` ветку GitHub.
2. GitHub Actions:
   - запускает тесты и линтер,
   - собирает Docker-образы (Next.js, NestJS, Worker),
   - пушит в GitHub Container Registry;
3. На сервере `watchtower` или webhook обновляет контейнеры через `docker compose pull && docker compose up -d`;
4. Zero-downtime deploy через rolling update (Nginx балансирует между старым и новым контейнером).

## 4. 24/7 доступность — как ничего не падает

### 4.1. Самовосстановление (auto-healing)
- Все сервисы — **Docker-контейнеры** с `restart: always`;
- **Health checks** на каждом контейнере (HTTP `GET /health` раз в 30 сек);
- если health check не прошёл 3 раза подряд — контейнер пересоздаётся;
- на уровне ОС — **systemd** следит, чтобы Docker был запущен.

### 4.2. Мониторинг
- **Uptime-монитор** (BetterUptime / UptimeRobot) — пингует `/health` каждую минуту из 5 регионов;
- **Sentry** — ловит необработанные исключения frontend и backend;
- **Grafana + Prometheus** (или Grafana Cloud) — метрики: CPU, RAM, disk, latency запросов, длина очереди PDF;
- **Алерты** (Telegram / e-mail / Slack) супер-админу при:
  - падении сервиса,
  - очереди PDF > 50,
  - диск заполнен на 85%,
  - ошибки БД.

### 4.3. Резервное копирование

**PostgreSQL:**
- **WAL-archiving** (continuous) в off-site S3 → позволяет восстановиться на любую секунду за последние 30 дней;
- **Ежедневный `pg_dump`** в 03:00 UTC → хранится 90 дней;
- **Еженедельный полный снапшот** диска VPS;
- Проверка восстановления — раз в месяц **автоматически** (тестовый restore в staging).

**MinIO / PDF:**
- **Версионирование файлов** включено;
- **Репликация бакета** в off-site S3 (реального-времени);
- Срок хранения PDF — **бессрочно** (требование ТЗ).

**Redis:** данные в Redis — кеш и очередь, при потере не критично; но включаем AOF-персистентность с синком каждую секунду, чтобы очередь PDF не терялась при рестарте.

### 4.4. SSL / сертификаты
- **Let's Encrypt** через Caddy или certbot;
- автопродление, алерт при ошибке;
- HSTS включён.

### 4.5. DDoS / защита
- **Cloudflare** перед всей системой: WAF, rate-limit, bot-protection;
- дополнительно на Nginx — rate-limit по IP (500 req/min на эндпоинт `/api/auth/*`).

### 4.6. План аварийного восстановления (DR)

| Сценарий | Reaction | RTO* | RPO** |
|---|---|---|---|
| Упал контейнер | Auto-restart | < 1 мин | 0 |
| Упал VPS | Перезапуск, проверка | < 15 мин | 0 |
| Потеря VPS (диск умер) | Восстановление из snapshot + WAL | < 2 ч | < 1 мин |
| Потеря региона провайдера | Поднятие в другом регионе из off-site backup | < 8 ч | < 1 ч |
| Случайное удаление данных | Point-in-time restore из WAL | < 1 ч | < 1 мин |

_* RTO — сколько времени сервис недоступен._
_** RPO — сколько данных теряется (в худшем случае)._

### 5.2. Ключевые ограничения (constraints)

```sql
-- Уникальность номера инвойса
ALTER TABLE invoices ADD CONSTRAINT uniq_invoice_number UNIQUE (number);

-- Статус только вперёд (через триггер)
CREATE FUNCTION prevent_status_rollback() RETURNS trigger AS $$
BEGIN
  IF OLD.status = 'issued' AND NEW.status = 'draft' THEN
    RAISE EXCEPTION 'Cannot move from issued back to draft';
  END IF;
  IF OLD.status IN ('paid', 'cancelled') AND NEW.status <> OLD.status THEN
    RAISE EXCEPTION 'Cannot change final status';
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Каскад
ALTER TABLE invoice_items ADD FOREIGN KEY (invoice_id)
  REFERENCES invoices(id) ON DELETE CASCADE;

-- Receipt обязан ссылаться на оплаченный инвойс
ALTER TABLE invoices ADD CONSTRAINT receipt_has_parent
  CHECK ((type = 'receipt' AND parent_invoice_id IS NOT NULL)
      OR (type = 'invoice' AND parent_invoice_id IS NULL));
```

### 5.3. Индексы — что и зачем

| Индекс | Назначение |
|---|---|
| `invoices.number` (UNIQUE) | Защита от дублей + быстрый поиск по номеру |
| `invoices.status, issue_date` (B-Tree) | Списки «все issued за период», фильтры в UI |
| `invoices.counterparty_id` | Все инвойсы конкретного контрагента |
| `invoices.our_company_id, issue_date` | Отчёт по компании |
| `invoices.parent_invoice_id` | Найти receipt по инвойсу |
| `invoice_items.invoice_id` | Загрузка позиций |
| `counterparties.name gin_trgm_ops` (pg_trgm) | **Поиск по подстроке** с поддержкой тайского — для type-ahead |
| `audit_log.entity, entity_id, created_at` | История изменений одной сущности |
| `audit_log.created_at` (BRIN) | Выборка по периоду |

### 5.4. Миграции (Prisma)
- Все изменения схемы — через `prisma migrate` (SQL-миграции под версионным контролем в git);
- Каждая миграция в dev → review → staging → production;
- Prisma генерирует типизированный клиент → компилятор ловит изменения схемы на этапе сборки.

### 5.5. Пример запроса (как приложение достаёт инвойс)

```typescript
// NestJS через Prisma
const invoice = await prisma.invoice.findUnique({
  where: { id: invoiceId },
  include: {
    ourCompany: { include: { bankAccounts: true } },
    counterparty: true,
    items: { orderBy: { positionNo: 'asc' } },
    ourBankAccount: true,
    receipt: true,   // если есть связанный receipt
  },
});
```
Одним запросом подтягивается вся карточка с вложенными сущностями → быстрый рендер preview.

### 5.6. Connection pooling
- **PgBouncer** перед Postgres в режиме `transaction` pooling;
- лимит соединений с приложения: 20, в PgBouncer: 50, в Postgres: 100 — чтобы pgbouncer сглаживал пики нагрузки;
- Prisma из коробки работает поверх PgBouncer.

### 5.7. Бэкапы БД (детально)

| Что | Где | Частота | Срок хранения |
|---|---|---|---|
| WAL-архив (Write-Ahead Log) | Off-site S3 (Backblaze B2) | Continuous (каждые 60 сек) | 30 дней |
| `pg_dump` (логический) | Off-site S3 | Ежедневно, 03:00 UTC | 90 дней |
| Snapshot диска VPS | У провайдера | Еженедельно | 4 снапшота |
| Логический дамп схемы | git-репозиторий | При каждой миграции | Вечно |

**Тестирование restore:** раз в месяц автоматически — берём последний `pg_dump`, разворачиваем в staging, прогоняем smoke-тесты. Если ломается — алерт.

### 5.8. Репликация (для HA)
- **Hot standby PostgreSQL** в другой зоне провайдера (streaming replication);
- при падении primary — failover в течение 1–2 минут (Patroni / автоматический promote);
- **на MVP репликация не обязательна** (8 компаний, небольшой объём), но архитектурно закладываем: `DATABASE_URL` — одна переменная, переключение на standby — смена DNS.

### 5.9. Объём данных — оценка

За 3 года при 50 инвойсах в месяц:
- invoices: ~1,800 записей × ~2 KB = ~3.6 MB
- invoice_items: ~5,400 × ~500 B = ~2.7 MB
- audit_log: ~100,000 × ~300 B = ~30 MB
- PDF-файлы: ~3,600 × ~150 KB = ~540 MB

**Итого БД — несколько сотен мегабайт, PDF-хранилище — меньше гигабайта.** Сервера 2 vCPU / 4 GB RAM хватит с огромным запасом на годы вперёд.

## 6. Безопасность

### 6.1. Сеть
- VPS в **private network** у провайдера: Postgres, Redis, MinIO слушают **только приватный интерфейс**;
- Публично доступен **только Nginx** (порты 80/443);
- SSH только по ключам, fail2ban.

### 6.2. Данные
- Пароли пользователей — **argon2id**;
- API-ключи Wise, секреты Auth.js — в env-переменных, никогда в git (1Password / Doppler для хранения);
- Шифрование БД-диска на уровне провайдера (LUKS у Hetzner);
- HTTPS для всего, HTTP → редирект на HTTPS.

### 6.3. Приложение
- **RBAC** на уровне NestJS guards — все эндпоинты покрыты;
- Prisma — параметризованные запросы, SQL-инъекции исключены;
- CSRF-токен для мутирующих запросов;
- CSP-заголовки + helmet.js;
- Логирование всех чувствительных действий в `audit_log`.

### 6.4. Доступ разработчиков
- GitHub-организация с 2FA-требованием;
- Доступ на prod-сервер — только через bastion-хост + SSH-ключи;
- Секретные переменные — через GitHub Secrets, никогда в коде.

## 10. Итого — почему эта архитектура

1. **Проста** — все компоненты в Docker, один `docker-compose.yml`, 1-2 VPS. Любой девопс поднимет.
2. **Надёжна** — авто-рестарт, репликация, бэкапы в 3 места, мониторинг 24/7.
3. **Дёшево** — от €22/мес в проде, масштабируется горизонтально при росте.
4. **Восстановима** — worst-case RTO 8 часов, RPO 1 час. Любая потеря данных ≤ 1 минуты.
5. **Расширяема** — Telegram-бот, Google OAuth, новые шаблоны инвойсов — всё вписывается без переделки архитектуры.
6. **Безопасна** — приватная сеть, HTTPS, argon2, RBAC, audit log всех действий.

На вашем объёме (9 компаний, до 100 инвойсов в месяц) эта конфигурация переживёт рост в 100× без единого изменения.
